Practice Runs

A mobile-first tool for the Thursday/weekend crew to set weekly availability and see when enough people overlap to actually get a run together. Replaces the Google Sheets prototype once Phase 1 ships.

Overview

Started as a Google Sheets prototype to validate the data model fast, with zero build time. It worked, but scaling it to time-of-day granularity (Anytime / Specific hours / Unavailable, per-day time ranges, notes, per-player overrides) pushed the grid past what Sheets — especially Sheets on mobile — can present cleanly. The core logic is proven; this doc is the plan for rebuilding the same logic as a small, purpose-built app with real interaction design instead of a spreadsheet grid.

Single job of the app: let anyone in the group see, at a glance, which day this week has the most players free and what hours actually overlap — and let them update their own (or someone else's) availability in a couple of taps, from a phone, without an account.

What pulling real group chat transcripts changed: the recurring weekly grid answers "when is everyone generally free" — but it turned out that's not the actual bottleneck. The real friction, visible directly in how the group already coordinates, is proposing a specific one-off session (rent a gym, hit an open-gym window, or meet at a park), tracking who's actually in against a headcount that matters for cost, and splitting the bill — all currently done by copy-pasting a numbered list in a group chat. That's captured below as Gap analysis and a full Session/Venue data model, not treated as a someday nice-to-have. The two features (recurring availability, one-off sessions) don't share internal state, so the plan built the grid first to validate the core tap-to-edit interaction, then layered Sessions on top without rework — see Build phases for why that was a sequencing choice, not a scope cut. Sessions & Venues (Phase 3) has since shipped: propose/edit/delete a session, RSVP in/out, and a live cost-split + headcount check for rented-gym sessions, all rendered directly below the grid on the same page.

Team: Uncrowned Kings. Logo palette (teal, red, gold/amber, black) happens to line up closely with the app's existing dark-navy/amber/teal theme already used throughout this doc — worth carrying that through into the real UI rather than picking a separate palette.

Decisions log

Identity
Name-based, remembered per device

First visit: pick your name from the roster. Stored client-side (localStorage or a cookie), no password, no account. Trust-based — anyone can edit anyone's row, same as the sheet. Editing "for" someone else is just tapping their row directly.

Save behavior
Optimistic, no confirmation screen

Tapping Save updates the grid cell instantly (before the network call resolves), then closes the drawer. The updated cell is the confirmation — no toast, no modal. On failure, the cell reverts and shows a small inline error.

Sync
Manual pull-to-refresh

No websockets, no polling. Refetch on pull-to-refresh and on page focus. Right-sized for ~15 people coordinating pickup runs — revisit only if the group actually complains about stale data.

Editing target
One grid, toggle changes what Save writes to

"Usual" and "This week" aren't separate pages — same grid, same tap interaction. The toggle only changes whether a tap writes to DayDefault or DateOverride. Cells inherited from Usual (no override yet) render faded; a cell with a real override gets a small dot marker.

Venues
Open to any player — add, edit, or delete, no approval step

Adding a venue (name, address, cost, hours) was originally admin-only, routing every suggestion through TJ — a real bottleneck for a group where players often already have the address/cost info themselves. Reversed: any player can add, edit, or delete a venue directly, matching the trust model every other write in this app already uses. Deleting a venue that still has sessions attached is blocked with an error rather than silently orphaning that session's venue reference.

Auth
Deferred, not built for now — and not the fix for portfolio exposure

Real accounts (Google/email via NextAuth) don't get built until there's a concrete trigger: opening this up to people outside TJ's direct trust circle. Note: auth alone wouldn't even solve "a recruiter could edit my real team's data" — that needs per-team permission logic on top, which is real added scope. See Portfolio & demo mode below for the actual fix to that problem.

Portfolio
Separate demo team, not a login wall

A second seeded Team row (/team/demo) with fake players, reset daily via a scheduled job. The real team's URL is simply never posted publicly — only /team/demo goes on GitHub/LinkedIn. Solves the actual concern (a recruiter clicking around and breaking real data) without auth, permissions, or delaying Phase 1. A visible "You're viewing a demo — data resets daily" banner is a cheap belt-and-suspenders addition.

Min players
Open question — needs a real team conversation, not a guess

$100 split 1 way isn't happening; split 10 ways is easy. Where the actual threshold sits is a team decision (8? 10?), not something to hardcode. Session.minPlayers stays editable per session rather than defaulted, so it's set once the number's agreed on and can vary by venue if cost differs.

Onboarding
Shipped — scripted guide-to-tap tour, dismissible, persisted the same way as identity

A short tour shown once, right after a brand-new player's first name pick, five steps in causal order — Usual Schedule (the player's own Monday cell, they tap it, the real edit drawer opens, they save it themselves), the This Week toggle (real tap), an inherited-vs-overridden explanation (the same Monday cell, now shown faded), the Team Window readout, and Games & practices (Game Day + Sessions) — so a brand-new player doesn't have to ask the group chat what to do. An earlier ordering taught mechanics out of sequence (created a This Week override before ever explaining Usual Schedule) and never mentioned Sessions/Game Day at all; reordered after trying it live so each step only depends on what came before. Resolved in favor of this scripted, guide-to-tap approach over a coach-marks library (react-joyride/driver.js): since every player's row starts all-UNAVAILABLE, having them actually perform the tap/save themselves doubles as their first real availability entry, not just a passive lesson — and it avoids a new dependency plus re-skinning a third-party library's default look to match this app's theme. A "Step X of 5" indicator and "Skip tour" appear at every step. Dismissal (skip or the final "Got it") writes a hasSeenTour flag in localStorage, mirroring how identity is already stored: no accounts, no DB table, no server round-trip.

Team window
Single swipeable card, not a per-day row in the grid

Phase 2 shipped the team window as a row of tiny per-day boxes inside the grid table — it worked, but was hard to read at mobile widths (the primary use case) and wasn't interactive. Replaced with a standalone card — closer to this doc's original mockup below — that defaults to the best-availability day and pages through the rest via always-visible arrow buttons, native touch swipe, and tappable dot indicators. A static single-best-day card (the mockup taken literally) was considered and rejected: it would hide the other 6 days' data the app already computes for free. The underlying calculation is unchanged — this is a display-only change.

Reset to Usual
Explicit undo for an override, not just re-editing it

Once a player explicitly overrides a This Week day, there was no way back to "just inherit Usual" short of manually re-entering Usual's exact values — easy to get subtly wrong. A DELETE .../override?date=YYYY-MM-DD endpoint removes the DateOverride row outright (idempotent, no-op if nothing to delete), and the edit drawer shows a "Reset to Usual" button whenever the open cell is already overridden. This is the correct semantic undo — it matches the existing invariant that a missing override row means "use the default," rather than introducing a separate "unset" state to reconcile.

Grid markup
ARIA grid pattern over a semantic <table>

The original <table>/table-fixed layout couldn't give per-breakpoint column sizing once the demo roster grew to a realistic ~15 rows (matching the real team's size, up from an earlier 8-player stub). Re-implemented as CSS Grid with explicit role="table"/"row"/"columnheader"/"rowheader"/"cell" attributes — same screen-reader table semantics, more layout flexibility.

Sessions page
No dedicated Sessions route — shares the grid page

SessionsView renders directly below the availability grid on the existing /team/[slug] page, instead of a separate /team/[slug]/sessions route. The two features share a device/identity context but not data; splitting into a second route would just add a navigation step for a single-page mobile app with room for both.

Venue management
Server Action pages, not POST/PATCH/DELETE API routes

/venues (list) + /venues/new (create) + /venues/[venueId]/edit + /venues/[venueId]/delete (createVenue/updateVenue/deleteVenue Server Actions), open to any player: no client outside these forms ever needs a typed JSON contract for venue writes, so Server Actions skip maintaining one nothing else calls.

Session ownership
UI-gated Edit/Delete, not server-enforced

Session.proposedById records who proposed it; the Edit and Delete buttons only render when proposedById === currentPlayerId, but the PATCH/DELETE endpoints don't check this server-side. Matches the existing trust model — any write to DateOverride already trusts the caller's playerId — rather than partially bolting on enforcement for one endpoint.

Team window calculation

Same overlap logic proven in the spreadsheet, moved server-side: for each day, take every player who is not Unavailable, treat "Anytime" as 00:00–23:59, then teamWindow = [MAX(all fromTimes), MIN(all toTimes)]. If the resulting window is invalid (max ≥ min), show "No common time" rather than a broken range.

Data model

Same shape as the spreadsheet's three tabs (Usual Schedule, This Week, Effective This Week) — normalized into tables, with the "Effective" view computed on read instead of stored.

// schema.prisma

model Team {
  id       String    @id @default(cuid())
  slug     String    @unique   // the shareable URL, e.g. "uncrowned-kings"
  name     String
  players  Player[]
  sessions Session[]         // live from Phase 3
}

model Player {
  id         String         @id @default(cuid())
  teamId     String
  team       Team           @relation(fields: [teamId], references: [id])
  name       String
  number     Int?           // jersey number, nullable — 0 is a valid value — live from Phase 1
  defaults   DayDefault[]
  overrides  DateOverride[]
  rsvps      Rsvp[]         // live from Phase 3
}

enum Status {
  ANYTIME
  SPECIFIC
  UNAVAILABLE
}

model DayDefault {
  id         String   @id @default(cuid())
  playerId   String
  player     Player   @relation(fields: [playerId], references: [id])
  dayOfWeek  Int      // 0=Sun ... 6=Sat
  status     Status   @default(UNAVAILABLE)
  fromTime   String?  // "18:00", only set when status = SPECIFIC
  toTime     String?
  note       String?  // e.g. "church," "work until 4:30" — live from Phase 1

  @@unique([playerId, dayOfWeek])
}

model DateOverride {
  id         String   @id @default(cuid())
  playerId   String
  player     Player   @relation(fields: [playerId], references: [id])
  date       DateTime // specific calendar date, not a day-of-week
  status     Status
  fromTime   String?
  toTime     String?
  note       String?  // e.g. "church," "work until 4:30" — live from Phase 2

  @@unique([playerId, date])
}

// --- Phase 3, shipped. ---
enum VenueType {
  RENTED_GYM        // booked and paid for, team has exclusive access, worth it only above minPlayers
  OPEN_GYM          // public gym time, no booking, show up and play as a team ("got next")
  PARK              // free, no booking, open anytime
  RECREATION_CENTER // public rec center, no booking, same cost-gating as OPEN_GYM/PARK
}

model Venue {
  id            String      @id @default(cuid())
  name          String
  type          VenueType
  address       String?
  bookingUrl    String?
  costPerHour    Int?       // cents — hourly rate, RENTED_GYM only
  openTime      String?     // "HH:MM" — informational hours-of-operation, not validated against Session times
  closeTime     String?     // "HH:MM"
  sessions      Session[]
}

model Session {
  id           String       @id @default(cuid())
  teamId       String
  team         Team         @relation(fields: [teamId], references: [id])
  venueId      String?
  venue        Venue?       @relation(fields: [venueId], references: [id])
  proposedById String?      // playerId of the proposer; null for legacy rows — gates Edit/Delete client-side only
  date         DateTime
  fromTime     String
  toTime       String
  costTotal    Int?         // cents — RENTED_GYM only, null for OPEN_GYM / PARK
  minPlayers   Int?         // RENTED_GYM only — worth booking only above this count
  rsvps        Rsvp[]
}

model Rsvp {
  id         String   @id @default(cuid())
  sessionId  String
  session    Session  @relation(fields: [sessionId], references: [id])
  playerId   String
  player     Player   @relation(fields: [playerId], references: [id])
  status     Status   // reuse ANYTIME as "in", UNAVAILABLE as "out"

  @@unique([sessionId, playerId])
}

Every day defaults to UNAVAILABLE — nobody is assumed free until they say so (carried over directly from the spreadsheet fix). DateOverride rows only exist when someone has actually overridden a day; a missing row means "use the default," which is exactly the fallback the Effective view computes.

Architecture

Stack

  • Next.js (App Router)
  • Prisma + Neon (Postgres)
  • Tailwind for styling
  • Deployed to Vercel
  • No auth library — identity is client-stored, no NextAuth needed in V1

Client state

  • Current player identity → localStorage
  • Grid data → fetched on load, refetched on pull-to-refresh / focus
  • Optimistic cell updates → local React state, reconciled after PATCH resolves

Future vision (not committed, not scoped)

If the concept proves out through real use with this team, a plausible next step is opening it up as a small multi-tenant tool — teams register their own account, add their own players and venues, run their own Usual/This Week/Sessions independently. This is explicitly a someday, if it's actually useful idea, not a roadmap commitment. It depends entirely on Phase 5's Auth trigger being met first, and on this actually solving a real problem for people beyond TJ's own team.

Out of scope (for now)

Stat sheet consolidation. There's a separate spreadsheet a player maintains for stats. Eventually it'd be nice to have one all-inclusive team hub, but folding that in now risks sprawling Phase 1 before it ships. Revisit after Practice Runs is actually in use.

Real-time sync, accounts/auth, push notifications. Explicitly decided against for the current group size and use case — see Decisions log.

Lessons from the spreadsheet prototype

  • Blank cells referenced across sheets evaluate to numeric 0, not empty string — broke the overlap MAX/MIN math and made valid "0" jersey numbers indistinguishable from "unset." Worth remembering if any server logic ever imports/exports sheet-shaped data.
  • Google Sheets data validation dropdowns don't reliably import when pointed at a text-only named range (only real cell-range references survive import consistently).
  • Merged cells crossing a frozen-pane boundary cause real scroll/selection glitches, especially on mobile Sheets — a preview of why a wide multi-field-per-day grid is fundamentally a poor fit for the spreadsheet medium once granularity goes up.
  • The single biggest friction point wasn't data entry — it was navigation: people landing on a read-only tab with no obvious path to editing their own row. The app's tap-to-edit-in-place design exists specifically to remove that gap.
