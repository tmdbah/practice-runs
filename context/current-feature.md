# Current Feature: Game Day

## Merge Target

main

## Status

<!-- Not Started|In Progress|Completed -->

Completed

## Goals

<!-- Goals & requirements -->

Add a "Game Day" concept alongside the existing practice-session Sessions feature, surfaced from a real group-chat gap: games are fixed (league schedule, known venue, never rented) and the weekly coordination need is a forfeit-risk headcount check ("need at least 5 in, or we forfeit"), not cost-split math. Full implementation plan: `/Users/tmdbah/.claude/plans/i-have-somewhat-of-lazy-wren.md`.

- Reuse the existing `Session`/`Rsvp` model with a new `kind` discriminator (`PRACTICE` | `GAME`) rather than a parallel `Game` model — mirrors how `SessionStatus` already extends `Session` with an enum.
- New `/team/[slug]` UI: a "Game Day" section above the existing "Sessions" section, each with its own "+ Propose Game" / "+ Propose Session" button, sharing one propose/edit form (kind set by which button opened it, not a manual toggle).
- Cost UI (cost/person, cost-if-N-join) is strictly hidden for `GAME` kind regardless of venue type — games are never rentals. Headcount (`minPlayers`) stays relevant for games but means "minimum to avoid forfeit," not "minimum to make a rental worth it" — distinct, more urgent red styling/copy ("Need N more to avoid forfeit") vs. the existing neutral amber wording for practice sessions.
- `minPlayers` defaults to `5` (the team's real forfeit threshold) when proposing a new Game, still editable; practice sessions keep no default.
- "Mark as Booked" is hidden for `GAME` kind (nothing is booked); "Cancel" stays available for both kinds via the existing `/confirm`/`/cancel` endpoints, unchanged.
- `kind` is set at creation only, not editable via the edit form — fixing a mis-proposed kind means delete + re-propose.
- Manual propose-one-at-a-time only, exactly like sessions today. No bulk/season-schedule import (explicitly deferred, see Notes) and no "one game per week" scheduling constraint (bye weeks/doubleheaders are real).

## Notes

<!-- Any extra notes -->

Bulk/season-schedule import (the user already has the full season's dates) was explicitly discussed and deferred — captain adds each week's game manually via "+ Propose Game," same as sessions today. Revisit only if that manual flow proves to be real friction once used.

Cost gating is strictly on `kind !== "GAME"` (not just venue type) even though the real game venue will be seeded as `OPEN_GYM` — nothing stops someone from picking a `RENTED_GYM` venue while proposing a game, so gating on kind (not venue) keeps cost UI from ever leaking through for games, matching the user's stated intent that games never involve a cost split.

Touches `prisma/schema.prisma` (new `SessionKind` enum + `Session.kind` field) and multiple `src/components/Sessions*.tsx` files — required a `/plan` pass before implementation per `CLAUDE.md`'s schema-file and >3-files rules (done, plan approved, see the plan file linked above).
