# Current Feature: Open Venue Submission & Venues Page

## Merge Target

main

## Status

<!-- Not Started|In Progress|Completed -->

Completed

## Goals

<!-- Goals & requirements -->

Open venue creation/editing up to all players (no more admin-only bottleneck), give venues a real browsable home instead of a hidden route, and add the small info gaps (hours, delete) that come with more entries from more people.

- Rename `/admin/venues` → `/venues` and `/admin/venues/new` → `/venues/new`; drop the "admin-added only" framing everywhere it's documented. No server-side gate is being removed — there never was one (the Server Action was already unguarded, just unlinked) — this is a discoverability/docs change, not a new trust boundary.
- Add a venue **edit** action (Server Action, no ownership/auth check — matches the existing DateOverride/Session/Rsvp trust model).
- Add a venue **delete** action (same, no ownership check).
- Add a "Venues" link on `/team/[slug]`, positioned to the right of the existing "+ Propose" session button, linking to the `/venues` list page (not an inline add-form on the team page).
- Schema: add optional `openTime` / `closeTime` string fields to `Venue` (HH:MM, same convention as existing time fields) — purely informational hours-of-operation, not validated against session propose times.
- Session propose/edit form's venue `<select>` shows richer per-option text (e.g. "INSZN — Rented Gym") instead of bare venue name.
- Explicitly **not** doing: Google Places API / address autocomplete (deferred — plain text address field stays as-is), any change to `VenueType` enum (no `OTHER` yet), any change to `Venue.costPerSession` semantics (stays a "typical/reference" value — `Session.costTotal`, set per-proposal, remains the real number driving cost-split math).

## Notes

<!-- Any extra notes -->

This deliberately reverses two previously-documented decisions: "Venues: Admin-added only, not open submission" (Decisions log) and the `/admin/venues` Server-Action-over-POST-route rationale (Architecture §7) — both discussed across several turns with the user and explicitly approved, not an oversight. Docs to reconcile: `AGENTS.md` ("Do NOT: Open venue submission to all players" bullet + Architecture Decisions bullet), `practiceRuns-ProjectOverview.md` (Routes & API table, Decisions log "Venues" row), `practiceRuns-Architecture.md` (§7 decision log rows re: venue admin/ownership), `practiceRuns-ProjectPlan.html` (matching Decisions log cards).

Also touches `SessionsView.tsx` (dropdown labels) and wherever the "+ Propose" button lives (adding the sibling "Venues" link) — more than 3 files total, flagged per `CLAUDE.md`'s confirm-before-proceeding rule.
